    if d == nil || d.Flipper == nil {
        return "", fmt.Errorf("firmware_introspect: Flipper transport unavailable")
    }

        if _, err := d.Flipper.DetectCapabilities(); err != nil {
            return "", fmt.Errorf("re-detect failed: %w", err)
        }

    if err != nil {
        return "", err

    return string(b), nil

### C.2 ‚Äî FirmwareBand resolution

```go
func resolveBand(fork, version string) string {
    f := strings.ToLower(fork)
    v := strings.ToLower(version)
    switch f {
    case "", "flipper", "official":
        return "stock/" + majorMinor(v)
    case "momentum":
        // version string "mntm-dev-<sha>" vs "0.x" vs "mntm-<ver>"
        switch {
        case strings.Contains(v, "dev"):
            return "momentum/mntm-dev"
        case strings.HasPrefix(v, "0."), strings.HasPrefix(v, "mntm-0"):
            return "momentum/mntm-stable"
        default:
            return "momentum/latest"
        }
    case "unleashed":
        return "unleashed/" + majorMinor(v)
    case "xtreme":
        return "xtreme/" + majorMinor(v)
    case "roguemaster":
        return "roguemaster/latest"
    default:
        return f + "/" + majorMinor(v)
    }
}

func majorMinor(v string) string {
    // strip leading "v"/"V", keep "0.x" or "1.2" ‚Äî everything after
    // the second dot collapses into ".x"
    v = strings.TrimPrefix(strings.TrimPrefix(v, "v"), "V")
    parts := strings.SplitN(v, ".", 3)
    if len(parts) < 2 {
        return v
    }
    return parts[0] + "." + parts[1] + ".x"
}
```

## D. Crypto1 cipher contract

### D.1 ‚Äî Package surface (already scaffolded)

`internal/crypto1/crypto1.go` ships the architect skeleton with:

```go
type Cipher struct { state uint64 }
func New() *Cipher
func (c *Cipher) Init(key uint64)
func (c *Cipher) Crypt(input uint32) uint32
func (c *Cipher) EncCrypt(input, nr uint32) uint32
func Prng(from uint32, n int) uint32
```

All method bodies are stubs returning zero values. The wave-2
engineer fills them in; no API surface changes.

### D.2 ‚Äî Bit ordering contract

- Key: LSB of `key` maps to LFSR bit 0 (lowest). Bit 47 of `key`
  maps to LFSR bit 47 (highest).
- Plaintext: LSB-first. The first bit clocked through the filter
  is bit 0 of `input`.
- Keystream: LSB-first. Output has the same bit width as input.
- nR feedback: LSB-first parallel with input.

This matches the reference C impls (mfoc, mfcuk, mfkey32v2) ‚Äî
the wave-2 engineer can cross-check against any of them byte-
for-byte.

### D.3 ‚Äî Test vectors

| Vector | Input | Expected output | Source |
|---|---|---|---|
| `Init(0)` zero-key | `Crypt(0)` twice | `0, 0` | derivable from spec |
| `Init(0xFFFFFFFFFFFF)` | first 8 bytes of keystream | see mfoc selftest | `nfc-tools/mfoc/test/crapto1_test.c::test_ones` |
| `mfkey32` recovery | `uid=0xCAFEBABE nt=0x01020304 nr_enc=<derived> ar_enc=<derived>` | `key=0xA0A1A2A3A4A5` | `equipter/mfkey32v2/README.md` |
| `EncCrypt` nR feedback | published 4-byte exchange | pinned | `RfidResearchGroup/proxmark3/client/src/mifare/mfkey.c` |
| `Prng(0x01020304, 64)` | n=64 | `0xEEDE3D4A` | `nfc-tools/mfcuk/src/crapto1.c::prng_successor` self-check |

All vectors are in the public literature. The clean-reimpl
disclaimer (see ¬ßE) means the engineer CAN use these vectors
verbatim ‚Äî they are facts about the algorithm, not copyrighted
expression.

### D.4 ‚Äî Known pitfall: LFSR tap documentation

**The mfoc and proxmark3 sources describe the 48-bit tap set in
incompatible notations** (mfoc uses MSB-numbered taps, iceman
uses LSB-numbered). The wave-2 engineer MUST resolve this by
running the published test vectors through the Go impl and
iterating on the tap set until all four vectors agree ‚Äî do NOT
trust either source's tap-numbering convention blindly.

Recommended workflow:
1. Start from the mfkey32v2 single-file C impl (simplest).
2. Port Init / Crypt byte-for-byte into Go with LSB-numbered
   taps.
3. Run `TestMfkey32KnownAnswer` ‚Äî if it fails, flip tap ordering
   and re-run.
4. Once that vector passes, run the remaining three.
5. If any disagree, the discrepancy is in `EncCrypt` / `Prng`,
   not the core LFSR ‚Äî diff the suspect method against iceman
   proxmark3.

Budget 1-2 days for vector-driven debugging. This is the
"meaty" part of the port.

## E. License posture

Classification:
- `clean-reimpl` ‚Äî algorithm is in public literature; Go code
  is derived from papers + unit-test cross-checks against public
  vectors; not copied verbatim from any specific source file.
  AGPL-compatible.
- `partial-port` ‚Äî some symbolic translation of C logic needed
  (e.g. specific magic-number tables); flag for license audit in
  v0.5.x.
- `wrapper-needed` ‚Äî algorithm requires shelling out to a binary;
  defer to v0.6 or drop.

| Port | Classification | Notes |
|---|---|---|
| `firmware_introspect` | clean-reimpl | Pure parsing of a public CLI's `device_info` output. No algorithm borrowed. No license concern. |
| `internal/crypto1` (Cipher, Prng) | clean-reimpl | Algorithm derived from ESORICS 2008 + IACR 2008 papers. Test vectors are public facts. **DO NOT paste from mfoc/mfcuk/proxmark3 `.c` files** ‚Äî the Go engineer rewrites from the paper. Risk: Low. |
| `mfkey32_recover` | clean-reimpl | Algorithm is ~40 lines; the `mfkey32v2.c` single-file impl is a useful reading reference only. Risk: Low. |
| `mfoc_attack` (nested) | partial-port | The nested attack has more moving parts (sector-walk, PRNG-rollback) than Crypto1 proper. Engineer may need to match mfoc's loop structure to hit runtime parity. **Flag for v0.5.x audit.** Risk: Medium ‚Äî if the port looks too similar to mfoc source after completion, refactor to distinct control flow. |
| `mfcuk_attack` (darkside) | partial-port | Same reasoning as mfoc. Darkside algorithm is published (Nohl et al.) but the concrete table of filter-output weights lifted by mfcuk is a potential licensing artefact ‚Äî reimplement from the math, not the table. **Flag for v0.5.x audit.** |

## F. Acceptance criteria (per engineer commit)

Every engineer's final commit on `feature/v0.5-blackhat-expansion`
MUST pass these gates before they hand the worktree back:

1. **Build clean.** `CGO_ENABLED=1 go build ./...`
2. **Vet clean.** `go vet ./...`
3. **Lint clean.** `PATH="$(go env GOPATH)/bin:$PATH" task lint`
4. **Tests pass with race.** `CGO_ENABLED=1 go test -short -race ./...`
5. **Registry size test.** The engineer bumps
   `internal/tools/registry_size_test.go`'s `expected` const by
   exactly the number of new Specs they Registered. Running
   `go test -run TestRegistrySize -count=1 ./internal/tools/...`
   stays green.
6. **Registry coverage test.** Every new Spec's `Risk` matches
   the corresponding entry in `internal/risk/risk.go::toolLevels`.
   `go test -run TestRegistryCoverage -count=1
   ./internal/tools/...` stays green.
7. **Hardware harnesses (orchestrator only).** After all five
   engineering tasks land, the orchestrator runs the four
   regression harnesses against a real Flipper on
   `/dev/ttyACM0`:

   ```bash
   CGO_ENABLED=1 go build -o bin/promptzero ./cmd/promptzero
   bin/hwtest     --bin ./bin/promptzero --port /dev/ttyACM0
   bin/mifaretest --bin ./bin/promptzero --port /dev/ttyACM0 --emulate=false
   bin/webtest    --bin ./bin/promptzero --port /dev/ttyACM0
   bin/clitest    --bin ./bin/promptzero --port /dev/ttyACM0
   ```

   A non-zero exit from any harness blocks v0.5.0 tagging. None
   of the new code paths are on the hardware critical path
   (offline crackers, host-side scanners, introspection) so the
   regression signal is "existing tools still work".

**Failing acceptance leaves the worktree open.** The engineer
diagnoses and fixes; they do not skip the gate.
