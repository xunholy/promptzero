## Phase 1 — Architecture foundation (Architect's Theme A)

This is the **load-bearing theme** for v0.8. Every other capability rides on it.

### 1a. `Backend` interface (replace Deps god-bag)

```go
type Backend interface {
    Name() string
    Capabilities() any   // returns backend-specific capability struct
    Health() error       // for system_capabilities Spec
    Close() error
}

type Deps struct {
    // -- always-wired primitives --
    Flipper *flipper.Flipper
    Audit   *audit.Log
    Config  *config.Config

    // -- backends, registered at startup --
    backends map[string]Backend  // "marauder", "bruce", "faultier", ...

    // -- LLM-side facilities --
    Generator   *generate.Generator
    Vision      *vision.Analyzer
    // ... (the LLM bag stays separate from the hardware bag)
}

func (d *Deps) Backend(name string) (Backend, bool) { ... }
func (d *Deps) RequireBackend(name string) (Backend, error) { ... }
```

Migrate ~50 handlers from `d.Bruce`/`d.Faultier`/`d.BusPirate` direct dereferences to `d.RequireBackend("bruce")`. Existing `RequireMarauder` becomes a thin wrapper.

### 1b. Concurrency-aware workflow runner

Today `internal/workflows/runner.go:runPhase` is sequential. Add:

```go
type ConcurrentStep struct {
    Tool     string
    Args     map[string]any
    Requires []string  // resource locks: "flipper.subghz", "marauder.wifi"
}

type Phase struct {
    Sequential []Step
    Concurrent []ConcurrentStep  // run as errgroup, share locks
}
```

Workflows opt in by returning `[]Phase`. Existing 8 workflows stay sequential by default; new ones (evil_twin_fullcap, glitch_chip_dump) declare concurrent steps.

### 1c. `system_capabilities` Spec (uniform discovery)

Returns:
```json
{
  "flipper":   { ... existing firmware_introspect bitmap ... },
  "marauder":  { connected: true, dual_band: false },
  "bruce":     { connected: true, board_type: "esp32-c5", has_5ghz: true, has_zigbee: true },
  "faultier":  { connected: false },
  "buspirate": { connected: true, fw_version: "5.0" },
  "federation": [
    { prefix: "secsec", healthy: true, tools: 31 },
    { prefix: "pm3",    healthy: false }
  ],
  "host":      { os: "linux", docker_present: true, container_images: ["ghcr.io/penthertz/urh-ng"] }
}
```

### 1d. Risk classification derivation

`risk.go`'s 250-entry static map is mostly a duplicate of `Spec.Risk`. Make `risk.Classify(name)` consult the registered Spec first; static map becomes legacy-synonym fallback only. Pairs with reviewer's drift findings.

**Effort: 4-6 weeks.** Touches ~50 handlers + the workflow runner. Can be done incrementally — add Backend interface alongside the existing fields, deprecate, remove.

## Notable cross-cutting decisions to make early

1. **Should `RecoverFast` be debugged or removed?** Engineer A's honest finding was that Garcia §4 doesn't deliver in our LFSR encoding. The parallel-fallback architecture works but is a workaround. Decide: invest a sprint in the equipter/mfkey32v2 lookup-table approach, or accept current performance and remove `RecoverFast` to simplify.

2. **Is hardnested worth porting natively?** Container bridge ships today. ~2 kloc bitslice C reimpl is a multi-week effort. With Proxmark3 federation as the live alternative and crypto1 covering 1K/4K cases, native hardnested may not be worth it. Stay container-bridge.

3. **Federation-vs-native for HackRF?** HackRF has Go bindings. Native `internal/hackrf/` is the right call (low-latency control). PortaPack-specific apps (Mayhem) might get federated MCP later.

4. **GhostESP vs Bruce vs Marauder unification?** All three are ESP32 + serial menu. After 3 backends with similar shapes, a `internal/esp32backend/` shared core might emerge. Defer until pattern is stable across all 3.

5. **Is the dark-web "Private-Unleashed 2.0" firmware worth defending against?** Researcher A flagged it. Adding `subghz_rollback_detect` (capture-only) is socially defensible and gives operators a way to detect when one's been used. Skip the offensive equivalent.

## What NOT to do

From the team's findings, things to explicitly **not** pursue:

- **Mifare Plus EV2 SL3 / DESFire EV2/EV3 attacks** — no public PoC, AES-128 holds. Stay current with research, don't speculate.
- **Tesla UWB relay** — needs UWB radio hardware not in scope.
- **5Ghoul / SNI5GECT** — specialized 5G test gear, niche.
- **TETRA:BURST** — needs Motorola/Sepura HW knowledge; mark as "researchable, not implementable today."
- **Ubertooth One backend** — EOL Dec 2022, Sniffle is the modern replacement.
- **SubSpectra** — archived 2026-01-04.
- **Original GhostESP** (Spooks4576) — archived; use GhostESP-Revival fork.

## Strategic verdict (from Architect, lightly edited)

> The architecture is aging well at the Spec layer and accumulating debt at the integration layer. The Spec registry pattern is genuinely good — adding the 300th Spec is the same amount of work as adding the 230th. **mcpfed riding the same registry was an excellent decision.** What's accumulating debt is integration: five backends got plumbed in as five direct fields on Deps because each landing wave was time-pressed; nobody factored out the abstraction. v0.8's window to factor out the Backend interface is the right time — the pattern across 5 backends is now stable enough to abstract, and one more would force the abstraction under worse conditions.
>
> The single architectural change that unlocks the most v0.8 capability: **introduce the `Backend` interface + concurrency-aware workflow runner together.** Every emerging attack (evil-twin, glitch-and-dump, NFC relay, CAN replay) is *blocked* by the inability to express "two devices, one phase." Everything else on this list can wait a quarter; that one cannot.
